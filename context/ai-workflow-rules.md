# AI Workflow Rules

## Approach

Build DocTutor AI incrementally using a spec-driven workflow. The six
context files define what to build, how to build it, and the current
state of progress. Always implement against these specs. Do not infer
or invent behavior that is not defined here. When in doubt, write an
open question in `progress-tracker.md` rather than guessing.

## Scoping Rules

- Work on one feature unit at a time. A feature unit is a single
  vertical slice: one API route + its lib function + the UI that
  consumes it.
- Prefer small, verifiable increments. A unit is complete when it
  works end to end — not when the backend is done and the UI is
  "coming later."
- Do not combine unrelated system boundaries in a single step.
  Database schema work is separate from UI work. AI prompt work is
  separate from auth work.

## When to Split Work

Split an implementation step if it combines:

- AI pipeline changes AND UI changes (do AI first, verify output,
  then build the UI that consumes it)
- Multiple API routes that do not depend on each other
- Database schema changes AND feature logic that uses the new schema
  (schema first, verify migration, then logic)
- Any behavior that is not clearly defined in the context files

If you cannot describe the done state in one sentence, the scope is
too large — split it.

## Implementation Order (MVP Build Sequence)

Follow this order. Do not skip steps or build out of sequence.

1. Project scaffolding — Next.js 15, TypeScript, Tailwind, shadcn/ui,
   Convex schema, Clerk auth
2. Convex schema — define all tables and indexes before writing any
   queries or mutations
3. Docs crawl endpoint — Firecrawl integration, URL validation,
   markdown extraction
4. Roadmap generation — `generateObject` call with Zod schema,
   Convex write, roadmap page UI
5. Lesson generation — on-demand streaming generation, lesson page UI,
   content caching in Convex
6. AI tutor — `useChat` integration in lesson page, scoped system prompt
7. Progress tracking — quiz submission, lesson completion writes,
   progress display on roadmap
8. Dashboard — course list, resume state, delete course
9. Auth gates — protect all pages behind Clerk, redirect unauthenticated
   users to sign-in
10. Error states and loading states — every async operation has a
    loading state and a graceful error fallback
11. Usage caps and freemium limits — track course count per user,
    enforce limits in API routes
12. Polish and launch prep — responsive layout check, empty states,
    onboarding hints

## Handling Missing Requirements

- Do not invent product behavior not defined in the context files
- If a requirement is ambiguous, resolve it in the relevant context
  file before implementing
- If a requirement is missing entirely, add it as an open question in
  `progress-tracker.md` and do not build anything that depends on it
  until it is resolved

## Protected Files

Do not modify the following unless explicitly instructed:

- `components/ui/*` — shadcn/ui generated components. Use the CLI.
- `convex/_generated/*` — auto-generated by Convex. Never touch.
- `middleware.ts` — Clerk middleware. Only modify when explicitly
  changing auth rules.

## Keeping Docs in Sync

Update the relevant context file whenever implementation changes:

- Stack or dependency change → `architecture.md`
- New system boundary or folder convention → `architecture.md`
  and `code-standards.md`
- New color token or UI pattern → `ui-context.md`
- New code rule or file convention → `code-standards.md`
- Feature complete, in progress, or new open question →
  `progress-tracker.md`

## Before Moving to the Next Unit

All of the following must be true:

1. The current unit works end to end within its defined scope
2. No invariant defined in `architecture.md` was violated
3. `progress-tracker.md` is updated to reflect completed work
4. `npm run build` passes with zero TypeScript errors
5. No `any` types, `@ts-ignore`, or `eslint-disable` comments were
   introduced
6. New environment variables are documented with example values in
   `.env.example`
